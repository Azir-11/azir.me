## 目的

本文旨在你在“想搭建一个以 Markdown 为核心的站点”时，在三类方案中快速抉择：

1. 自定义组合（本文称“SSG方案”，总结参考自antfu老师的博客网站）：`Vite + unplugin-vue-markdown + unplugin-vue-router + vite-ssg`。
2. VitePress：官方维护的文档生成工具。
3. Nuxt：全功能应用框架（SSR + SSG + 模块生态）。

你将获得：

- 精简的比较维度（含“什么时候不要选”）。
- 可执行的决策树与问卷式选择器。
- 风险与迁移成本评估。
- 典型演进路径建议。

## 一分钟速览（Executive Summary）

| 选项      | 适配度                 | 最佳应用                              | 放弃它的信号                 |
| --------- | ---------------------- | ------------------------------------- | ---------------------------- |
| SSG方案   | 灵活度 100% / 维护自负 | 交互型文档、组件演示、内容 + 业务混合 | 团队缺少深度前端、维护人单一 |
| VitePress | 上线速度 & 稳定性      | 纯文档、API、轻量博客                 | 需要复杂交互或自定义构建链   |
| Nuxt      | 工程能力、SSR、生态    | SEO 严格、需动态数据、跨模块共享逻辑  | 内容以静态为主且无服务端逻辑 |

## 核心比较维度（Structured Comparison）

1. 灵活性

   - SSG方案：最大 —— Markdown 真正编译成 Vue SFC；你可自定渲染、生成资产（OG、索引、图像处理）。
   - VitePress：主题与配置窗口有限；可扩展但抽象层限制更高。
   - Nuxt：极高（页面、插件、中间件、模块体系），但对“纯内容”场景来说可能是超配。

2. 上手 / 认知负载

   - SSG方案：需要理解多个 unplugin 原理与 Vite 插件钩子。
   - VitePress：几乎零门槛；写 Markdown 即可。
   - Nuxt：框架理念 + 目录约定 + SSR 心智模型。

3. 性能（静态内容场景）

   - 三者都能达到不错的静态 Lighthouse 指标；差异主要来自：
     - 构建链自定义程度（SSG方案可做最细粒度优化）。
     - 运行时体积（VitePress 默认较轻；Nuxt 取决于启用的特性）。

4. SEO / SSR 需求

   - SSG方案：默认 SSG；SSR 需额外 glue（可接 `vite-plugin-ssr` / `Nitro`）。
   - VitePress：SSG 足够大多数文档；SSR 不主打。
   - Nuxt：原生 SSR + 混合渲染策略（按需动态）。

5. 生态与维护

   - SSG方案：依赖插件维护活跃度；你负责 glue 稳定。
   - VitePress：官方团队背书，升级节奏可控。
   - Nuxt：大型社区与模块生态；更新带来较大版本迁移注意事项。

6. 多人协作 / 交接成本

   - SSG方案：需要文档化（见 `md-to-vue-site.md`）；否则知识集中造成风险。
   - VitePress：新成员 10 分钟上手。
   - Nuxt：需培训 SSR / 路由 / 服务端模块能力。

7. 场景匹配度示例
   - 互动演示（代码沙箱 / 组件 Playground）：SSG方案 or Nuxt。
   - 单语种 API 文档：VitePress。
   - 营销站 + 登录区 + 博客共存：Nuxt。
   - 大量构建期生成资产（图像批处理 / 内容派生文件）：SSG方案。

## 决策树（Decision Tree）

回答下面问题，按第一次满足的路径选方案：

1. 你是否需要“运行时”动态数据（用户登录、个性化）且必须 SSR？
   - 是 → 选 Nuxt。
2. 你是否只需要“写文档 + 少量示例组件”并希望最少配置？
   - 是 → 选 VitePress。
3. 你是否需要在 Markdown 中大量嵌入真实业务组件 / 可视化交互 / 实验型演示？
   - 是 → SSG方案。
4. 你是否需要高度定制构建（批量生成 OG、内容索引、图像派生、多格式输出）？
   - 是 → SSG方案。
5. 否则：优先 VitePress（除非后期出现扩展瓶颈再迁移）。

## 选择问卷（Scoring）

为每条打分：1（无需求）~ 5（刚性需求）。将匹配最高的方案作为候选。

| 指标                               | SSG方案权重 | VitePress 权重 | Nuxt 权重 |
| ---------------------------------- | ----------- | -------------- | --------- |
| 深度自定义 Markdown 渲染           | 5           | 2              | 3         |
| 需要 SSR / 动态数据                | 2           | 1              | 5         |
| 纯内容交付 / 快速上线              | 2           | 5              | 2         |
| 组件交互演示密度                   | 5           | 3              | 4         |
| 团队前端经验平均值较低             | 1           | 5              | 2         |
| 自定义构建产物（OG、索引、批处理） | 5           | 2              | 3         |
| 长期低维护诉求                     | 2           | 5              | 3         |

统计：`方案得分 = Σ(实际需求分 * 权重)`。

## 各方案优缺点摘要

### SSG方案

优点：

- 灵活度极高：内容即组件；一切可插拔。
- 构建期扩展空间：任意 Hook/脚本注入。
- 运行时体积可控（剔除不需要的框架层）。
  缺点：
- 需要资深人员守护；抽象演进成本自负。
- 升级链路需自己回归测试。
- 缺少“默认主题”与统一导航样式，需要自行设计。

适合：开发型团队、设计系统文档、产品交互 Demo 合集、工程实验室。

### VitePress

优点：

- 最低学习曲线；官方文档质量高。
- 内置主题适配大多数文档站点。
- 维护简单，少量配置即可升级。
  缺点：
- 主题深度定制需要阅读源码或覆写结构。
- 内容与复杂业务代码耦合能力有限。
- 构建期 Asset 管线可玩性不高。

适合：快速输出、API 文档、知识库、个人/团队博客。

### Nuxt

优点：

- SSR / SSG / Edge 混合渲染一站式支持。
- 模块生态（Auth、i18n、Image、Content）完备。
- SEO 强诉求 & 动态数据场景成熟方案。
  缺点：
- 框架层开销（学习 + 运行）。
- 编译链更复杂，调试门槛高。
- 内容为主项目可能显得“重”。

适合：需要用户上下文、权限、服务端逻辑、复杂路由策略的业务站点。

## 迁移策略（When to Migrate）

| 原起点    | 触发因素                          | 建议迁移方向           | 迁移关注点                                          |
| --------- | --------------------------------- | ---------------------- | --------------------------------------------------- |
| VitePress | 需要高自由交互 / 自定义构建产物   | → SSG方案              | 逐步替换：先接入 `unplugin-vue-markdown` 再解耦主题 |
| VitePress | 需要 SSR / 登录 / 动态数据        | → Nuxt                 | 内容迁移到 `content/` 或自建 Markdown 管线          |
| SSG方案   | 维护人离职 / 人员流失风险         | → VitePress            | 减少自研脚本，保留最核心特性                        |
| SSG方案   | 新需求：完整服务端拼接 / API 聚合 | → Nuxt（或局部微前端） | 渐进式：先把动态区块独立 Nuxt 应用                  |
| Nuxt      | 渲染纯静态、性能优化难度高        | → SSG方案/VitePress    | 拆分：文档与应用分仓或同仓多包                      |

## 常见误区（Misconceptions）

1. “Nuxt 一定比 VitePress 慢” —— 错误。Nuxt 可通过分层缓存、边缘部署获得优秀性能，但纯静态 Markdown 用 Nuxt 是额外复杂度。
2. “SSG方案难以维护” —— 条件：若写清楚文档 + 脚本内聚 + 单元测试覆盖，其维护成本可控。
3. “VitePress 不能深度定制” —— 可以，但成本来自主题结构的理解与覆写，不是完全不可能。

## FAQ

Q：我能否先用 VitePress，未来再迁移到SSG方案？
A：可以。建议：保持 Markdown frontmatter 规范化、减少特定于 VitePress 主题的自定义组件耦合。

Q：SSG方案如何引入 SSR？
A：可结合 `vite-plugin-ssr` 或迁移到 Nuxt（保留内容部分管线），也可在 Node 层用 `@vue/server-renderer` 手写轻量 SSR。

Q：Nuxt 与SSG方案性能差异大吗？
A：对纯静态内容不显著；差别多来自构建体积与运行时初始化。Nuxt 更多的“基座”代码，但可通过模块裁剪与分层缓存优化。

Q：是否要一开始就做SSG方案？
A：除非你已确认需要高度定制（交互 Demo / 构建期生成资产 / 多维度扩展），否则建议“先简单后演进”。

## 总结

不存在“最好”的唯一方案，只有“成本与能力的最优匹配”。建议采用“最小可满足原则”：

1. 能用 VitePress 满足 80% 需求，就先用 VitePress；
2. 出现交互/构建自定义瓶颈，再迁到SSG方案；
3. 业务行为（认证、实时数据）驱动后端集成需求，再评估 Nuxt 或混合架构。
